## 一、測試方法

### 模擬使用者行為

每位虛擬使用者（VU）依序執行以下完整流程，模擬活動現場的真實操作：

| 步驟 | 動作 | 對應 API |
|------|------|----------|
| 1 | 登入（輸入姓名與身份） | `POST /api/login` |
| 2 | 載入參與者名單 | `GET /api/participants` |
| 3 | 輪詢目前發言者 | `GET /api/current-speaker` |
| 4 | 查詢個人媒合記錄 | `GET /api/connections` |
| 5 | 標記媒合（50% 機率觸發） | `POST /api/connect` |

### 測試參數

- **每階段持續時間**：45 秒
- **測試級距**：30 / 50 / 100 / 150 / 200 人同時上線
- **驗收標準**：錯誤率 0%、P95 < 500ms

## 三、結果分析

### 3.1 P95 穩定性

```
P95 回應時間趨勢：
30人  ██████████████████████████████████  137ms
50人  █████████████████████████████████   135ms
100人 █████████████████████████████████   137ms
150人 ████████████████████████████████████140ms
200人 █████████████████████████████████████148ms
```

P95 從 30 人到 200 人僅增加 **11ms**（137ms → 148ms），隨負載增加的劣化幅度極小，系統具備良好的線性擴展能力。

### 3.2 網路延遲基準

本機測試最低回應為 0.1ms，線上測試最低回應為 **43ms**，差距即為台灣至 Zeabur 節點的單次網路來回時間（RTT）。

扣除網路延遲後，伺服器本身的純處理時間：

| 人數 | 線上中位數 | 扣除 43ms 網路延遲 | 伺服器純處理時間 |
|------|------------|-------------------|-----------------|
| 100 人 | 55ms | 55 - 43 = **12ms** | 極快 |
| 200 人 | 58ms | 58 - 43 = **15ms** | 仍然極快 |

### 3.3 偶發性尖峰分析

50人、100人、200人測試中各出現一次 3～7 秒的 max 值，但 **P95 完全不受影響**，說明尖峰屬於孤立事件（影響比例 < 5%）。

可能原因：

| 原因 | 說明 |
|------|------|
| Zeabur 容器資源排程 | 雲端平台在高負載時偶爾進行內部資源重分配 |
| SQLite WAL checkpoint | WAL 日誌定期合併回主資料庫時產生短暫鎖定 |
| HTTPS 連線建立 | 新連線的 TLS 握手在高並發時可能排隊 |

## 五、結論與建議

### 結論

**系統通過所有壓力測試，可安全用於活動現場。**

- 全程錯誤率 **0%**，100 人使用情境下 P95 回應時間為 **137ms**，遠低於 500ms 驗收門檻
- P95 在 30～200 人區間僅增加 11ms，系統擴展性優秀
- 活動預計規模（100 人以內），系統有充足的效能餘裕

### 建議事項

| 優先度 | 建議 | 說明 |
|--------|------|------|
| 低 | 監控 WAL checkpoint | 若活動中出現偶發延遲，可調整 SQLite `wal_autocheckpoint` 參數 |
| 低 | 活動前重啟伺服器 | 確保容器處於熱狀態，避免冷啟動尖峰發生在活動進行中 |
| 參考 | 如未來擴展至 500人+ | 可考慮加入 `/api/current-speaker` 回應快取（TTL 3秒），進一步降低資料庫讀取壓力 |
